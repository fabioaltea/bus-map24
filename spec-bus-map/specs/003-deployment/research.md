# Research: Production Deployment

**Feature**: `003-deployment` | **Date**: 2026-04-24

---

## Decision 1: Frontend Hosting

**Decision**: Vercel  
**Rationale**: Vite SPA is static after `pnpm build`; Vercel has zero-config Vite support, instant CDN, free tier. `VITE_API_URL` env var already wired in `src/services/api.ts`.  
**Config needed**: `vercel.json` with `root = "bus-map-web"`, `buildCommand = "pnpm build"`, `outputDirectory = "dist"`, SPA rewrite rule.

---

## Decision 2: Backend Hosting

**Decision**: Railway  
**Rationale**: Single platform for API + worker + DB + Redis; GitHub integration; supports custom Docker services (required for PostGIS); persistent volumes for PMTiles; ~$15/mo for MVP.

---

## Decision 3: PostgreSQL + PostGIS

**Decision**: Railway custom service using `postgis/postgis:17-3.4` Docker image  
**Rationale**: Railway's managed Postgres doesn't include PostGIS. Running `postgis/postgis:17-3.4` as a custom Docker service is the standard approach. Costs same as a Railway database addon (~$5/mo).  
**Alternatives considered**:
- Supabase (PostGIS included, free tier) — requires external DB, adds latency between Railway API and Supabase
- Neon (PostGIS via extension) — serverless, cold starts hurt BullMQ workers
- Railway managed Postgres — no PostGIS, ruled out

**Required env vars on service**:
```
POSTGRES_DB=busmapdb
POSTGRES_USER=busmap
POSTGRES_PASSWORD=<secret>
```
`DATABASE_URL` for other services: `postgresql://busmap:<pw>@<internal-hostname>:5432/busmapdb`

---

## Decision 4: Redis

**Decision**: Railway Redis addon  
**Rationale**: Native Railway integration, shared private network with API and worker. Free tier 25MB, paid $1/mo.

---

## Decision 5: Worker entry point

**Decision**: Create `bus-map-api/src/worker.ts` — standalone process that imports and starts all BullMQ workers  
**Rationale**: Workers are defined (`catalog-sync.worker.ts`, `feed-download.worker.ts`) but never started. Server process should not run workers (separation of concerns, Railway allows separate services from same repo).  
**Deployed as**: Railway background worker service with `startCommand: "node dist/worker.js"`

---

## Decision 6: CORS

**Decision**: `CORS_ORIGIN` env var; defaults to `*` in dev, must be set to Vercel domain in prod  
**Rationale**: Current `origin: true` is too permissive for production. `@fastify/cors` already registered — just needs the `origin` option to read from env.

---

## Decision 7: Migrations on deploy

**Decision**: `railway.json` `deploy.startCommand` runs `node -e "import('./dist/db/migrate.js')"` before starting API, OR add a dedicated `migrate` script that runs `drizzle-kit migrate` via `tsx` in the build phase.  
**Rationale**: Drizzle Kit's `drizzle-kit migrate` needs `tsx` at runtime; simpler to add a `"db:migrate:prod"` script using the compiled migration runner.  
**Implementation**: Add `src/db/migrate.ts` that calls Drizzle's `migrate()` programmatically; compile it; run via `node dist/db/migrate.js && node dist/server.js` as the start command.

---

## Decision 8: PMTiles storage

**Decision**: Railway persistent volume mounted at `/app/tiles`  
**Rationale**: PMTiles are generated by `tile-gen.job.ts` and served via Fastify static. Railway volumes persist across deploys. 1GB volume ~$0.25/mo.  
**Alternative**: Cloudflare R2 — better for CDN but requires SDK changes; deferred to future feature.

---

## Decision 9: Monorepo config

**Decision**: Vercel project root = `bus-map-web`; Railway services use `rootDirectory` = `bus-map-api`  
**Rationale**: Both tools support monorepo root directory configuration without Turborepo or workspace tooling.
